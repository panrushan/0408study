## Key Concepts Covered

### 1. Git Stash（临时存档）
- **用途**：临时保存未提交的修改，切换分支后再恢复
- **核心命令**：
  - `git stash` - 保存当前进度
  - `git stash pop` - 恢复并删除 stash
  - `git stash list` - 查看所有 stash
- **场景**：代码写到一半，突然要切换分支修 bug

### 2. Git Rebase -i（历史整理）
- **用途**：合并多个提交为干净的提交历史
- **核心命令**：`git rebase -i HEAD~N`
- **操作方法**：第一个用 `pick`，后面的改成 `squash`
- **场景**：提交了多次"修改"、"又修改"，想合并成 1 个

### 3. Git Cherry-pick（挑拣提交）
- **用途**：把另一个分支的特定提交应用到当前分支
- **核心命令**：`git cherry-pick <commit-hash>`
- **场景**：只需要主分支上的某个 bug 修复，但不想合并整个分支

### Exercise 2: 历史整理术 🔧

**场景**：
> 你开发一个功能，连续提交了 3 次，但提交信息写得很随意。想合并成 1 个干净的提交。

**Question**：
1. 执行什么命令进入"整理模式"？
2. 进入编辑器后，怎么修改？

**User Answer**：
- 命令：`git rebase -iHEAD`（格式有误）
- 修改：第一个用 pick，后面改成 squash（不知道具体怎么改）

**Correct Answer**：
1. 正确命令：`git rebase -i HEAD~3`（注意空格和波浪号）
2. 修改方式：
   ```
   pick abc1234 改了点东西
   squash def5678 又改了
   squash ghi9012 修好了
   ```

**Explanation**：
- `-i` 和 `HEAD` 之间要有空格
- `HEAD~3` 表示最近 3 个提交
- 第一个保留 `pick`，后面的改成 `squash`（或简写 `s`）
- 保存后会弹出第二个编辑器写新的提交信息

**Result**：❌ 部分正确（命令格式有误，操作方法需加强）
